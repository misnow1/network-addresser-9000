# django-auditlog for the mutation audit trail

ADR 0005 named two candidate audit-history packages — `django-auditlog` and `django-simple-history` — without picking one. `django-auditlog` was chosen: its `register(model, include_fields=[...])` API maps directly onto ADR 0004's requirement to scope tracking to "operationally meaningful events... rather than every field on every object," so the fields that can be overridden or reassigned (`RackVlanRange.address_range`, `NetworkSwitchAddress.address`, `NetworkDevicePort.address`/`default_gateway`, `NetworkSwitch`/`NetworkDevice` `rack`/`rack_slot`) are named explicitly, while everything else on those models is left untracked. It also has a lighter schema footprint — one generic `LogEntry` table via signals, versus a full shadow table per tracked model with `django-simple-history`'s `HistoricalRecords()` — and `django-simple-history`'s per-version full-snapshot design is built around reverting to any prior version, a capability ADR 0004 never asked for. The one wrinkle: `include_fields` filters create/update/delete diffs uniformly, so a scoped model's delete log entry only shows the scoped field(s), not a full snapshot — acceptable because each `LogEntry`'s `object_repr` (the model's own `__str__`) is captured independently of `include_fields` and still identifies what was deleted.
